# Worked example: `trident-vm-testimage` mapped to the tailor image-definition mechanism

This is a **real, verified** mapping of trident's `tests/images/trident-vm-testimage` (base phase,
the `grub` / `root-verity` / `usr-verity` variants) into the proposed
[image-definition layout](../../image-definitions.md), extended with two thought experiments: a
`vm-img` variant (a different *output format*) and a **hypothetical, Fedora-aligned Azure Linux `4.0`
release**. Every authored fragment keeps **tailor fields at the top level and IC config under
`config:`**; the matrix logic lives in the **directory structure** (`by-variant/`, `by-arch/`,
`by-release/`), not in YAML directives. The directives in play are one `$include` per variant (shared
storage layout), a `$replace` (vm-img output), and a `$set` (4.0 base override) ‚Äî plus a couple of
`params` for package names that vary by release/arch.

> **Verified by `tailor`.** `tailor render trident-vm-testimage` composes every cell (tailor fields +
> `config:`, with `$include`/`$set`/`params` resolved) into [`rendered/`](./rendered), and a golden
> test (`reproduces_every_trident_golden`) re-renders all cells in CI and fails on any drift. Each
> **3.0** cell reproduces the corresponding original trident file (order-independent lists like
> `packages` as sets; order-sensitive lists like `kernelCommandLine`/`scripts` in order):
>
> ```
> PASS  grub         3.0 rendered == baseimg-grub.yaml
> PASS  root-verity  3.0 rendered == baseimg-root-verity.yaml
> PASS  usr-verity   3.0 rendered == baseimg-usr-verity.yaml
> ```
>
> The `4.0` and `vm-img` cells are **hypothetical** (no original to diff against); their
> fully-composed configs sit in [`rendered/`](./rendered) beside the 3.0 goldens, so you can see the
> "after" full file beside the small authored fragments.

> **Golden filenames are the cell slug** (`<image>_<each axis value in matrix order>_<format>`, `_`
> -separated ‚Äî design.md ¬ß10), which is also the output-artifact naming. Every cell is checked in and
> regenerated by `tailor render` (the golden test re-renders all 16 in CI and fails on drift). The
> **3.0** cells are arch-*independent*, so `‚Ä¶_amd64_3.0_‚Ä¶` and `‚Ä¶_arm64_3.0_‚Ä¶` are byte-identical; the
> **4.0** cells are arch-*dependent* (the grub EFI package name differs), so the two arches diverge ‚Äî
> both are visible side by side.

> **No `base.yaml`, no `axes.yaml`.** The shared IC config lives directly in `image.yaml` (the
> always-applied "base document"); the `matrix:` in `image.yaml` *is* this image's axis declaration.
> A separate repo-wide `axes.yaml` is optional and only earns its keep when many images must share one
> validated vocabulary.

## Mapping: original trident file ‚Üí authored files that compose it

| Original (`base/‚Ä¶`)        | Composed from                                                        |
| -------------------------- | -------------------------------------------------------------------- |
| `baseimg-grub.yaml`        | `image.yaml` (shared) + `by-variant/grub.yaml` + `layouts/storage/grub.yaml`         |
| `baseimg-root-verity.yaml` | `image.yaml` (shared) + `by-variant/root-verity.yaml` + `layouts/storage/root-verity.yaml` |
| `baseimg-usr-verity.yaml`  | `image.yaml` (shared) + `by-variant/usr-verity.yaml` + `layouts/storage/usr-verity.yaml`   |
| *(none ‚Äî hypothetical)*    | `image.yaml` (shared) + `by-variant/vm-img.yaml` + `layouts/storage/grub.yaml`               |

> **Thought experiment ‚Äî `vm-img` (renders to
> `rendered/trident-vm-testimage_vm-img_amd64_4.0_base_vhd-fixed.yaml`).** A plain bootable VM image
> that emits a **fixed-size VHD** instead of a cosi. It demonstrates that the *output format is a
> tailor field* (`outputs:`, driven by `--output-image-format`), so it sits at the **top level** ‚Äî
> cleanly separate from the IC config under `config:`. Because `outputs:` is a **list that append**s by
> default (so one image can emit several formats at once ‚Äî each `(cell √ó format)` is its own
> artifact), swapping cosi ‚Üí vhd-fixed uses `$replace`:
>
> ```yaml
> outputs: { $replace: [ { format: vhd-fixed } ] }   # instead-of, not in-addition-to
> ```
>
> It has **no original trident file** to verify against ‚Äî it's purely illustrative of the
> tailor-field-vs-`config:` split and the append-vs-replace choice.

On **3.0**, each variant reuses the *same* composed config across **amd64 and arm64** (exactly as
trident's `testimages.py` does ‚Äî the arm64 entries point at the same YAML); only the base image
differs, so it lives in `by-arch/`. That alone replaces trident's separate `*-arm64` image entries.
(On **4.0** the config becomes arch-*dependent*, because the Fedora-aligned grub package name is
arch-specific ‚Äî see below.)

## Adding a Fedora-aligned `4.0` release (the payoff)

Azure Linux **4.0** (in beta at `packages.microsoft.com/azurelinux/4.0/beta`) realigns many package
names to Fedora conventions. Adding the whole release was **one matrix line + one
`by-release/4.0.yaml`** ‚Äî no existing file was edited, and **every 3.0 cell stays byte-for-byte
identical** (verified). The real package-name divergences, checked against the live beta repo:

| Concern | 3.0 (AZL) | 4.0 (Fedora-aligned) | Where it's expressed |
| ------- | --------- | -------------------- | -------------------- |
| package manager | `dnf` | `dnf5` | `by-release/*.yaml` ‚Üí `config.os.packages.install` |
| firewall | `iptables` | `iptables-nft` | same |
| editor | `vim` | `vim-enhanced` | same |
| bootloader (verity) | `systemd-boot` | `systemd-boot-unsigned` | `sdBootPkg` param ‚Üí verity `install` |
| grub EFI (grub/vm-img) | `grub2-efi-binary` | `grub2-efi-x64` / `grub2-efi-aa64` | `grubEfiPkg` param (**arch-specific!**) |

Two patterns split the work by *how* a package diverges:

- **Shared, release-only renames ‚Üí `by-release/`.** `dnf`/`iptables`/`vim` appear in *every* variant
  and differ only by release, so they moved out of `image.yaml`'s shared `config:` into
  `by-release/3.0.yaml` and `by-release/4.0.yaml`. The release-invariant packages stay shared.
- **Variant-scoped names that also vary by release *and* arch ‚Üí `params`.** The grub EFI package is
  *installed* by `grub`/`vm-img` and *removed* by the verity variants, and on 4.0 its name is
  **arch-specific**. Repeating that name (and its release/arch logic) across three variant files would
  be copy-paste, so it is a single param `grubEfiPkg`, set per release and composed from an arch token
  `efiArch`:

  ```yaml
  # by-arch/amd64.yaml          (arm64.yaml sets aa64)
  params:
    efiArch: x64
  # by-release/3.0.yaml
  params:
    grubEfiPkg: grub2-efi-binary           # one arch-independent name
  # by-release/4.0.yaml
  params:
    grubEfiPkg: "grub2-efi-${efiArch}"     # ‚Üí grub2-efi-x64 (amd64) / grub2-efi-aa64 (arm64)
  # by-variant/grub.yaml        (root-verity / usr-verity reference it in `remove:`)
  config:
    os:
      packages:
        install: ["${grubEfiPkg}"]
  ```

  This **release √ó arch interaction** is the genuinely tricky case, and it is handled with one param
  and zero nested logic ‚Äî the clearest argument for params earning their keep
  ([mechanism ¬ß8](../../image-definitions.md)).

4.0 also ships from a **different source**: a beta `--rpm-source` repo
([`repos/azurelinux-4.0-beta.repo`](./repos/azurelinux-4.0-beta.repo), a `.repo` pointing at the
remote beta `baseurl`, referenced by the `rpmSources` tailor field) and an **MCR** base image instead
of 3.0's local vhdx (the `base:` `$set` in `by-release/4.0.yaml` overrides the by-arch default). Both
are **tailor fields at the top level** ‚Äî never under `config:`. The net effect is in the goldens:
`‚Ä¶_grub_amd64_4.0_base_cosi.yaml` installs `grub2-efi-x64`, `‚Ä¶_grub_arm64_4.0_base_cosi.yaml` installs
`grub2-efi-aa64`, and both swap `dnf`‚Üí`dnf5`, `iptables`‚Üí`iptables-nft`, `vim`‚Üí`vim-enhanced`.

## Metrics (honest)

- **Authored:** a folder of small single-purpose files ‚Äî the three genuinely-different storage layouts
  dominate; everything else (the shared `image.yaml`, the per-variant deltas, the by-arch/by-release
  fragments) is a handful of lines each.
- **Replaced:** the 3 original full 3.0 files (452 lines) ‚Äî **and** the arm64 near-duplicates trident
  maintains separately, **and** the entire 4.0 release that does not exist in trident yet.
- The real win isn't a line-count reduction on one family; it's that **the shared core is written
  once** and **the multiplying axes (arch, release) cost almost nothing**: we added a *divergent,
  Fedora-aligned 4.0* across all four variants and both arches with **one matrix line plus one
  `by-release/4.0.yaml`**, and every 3.0 golden stayed byte-identical.

## What decomposed cleanly ‚úÖ

- `os.bootloader`, `os.selinux`, `os.users`, the release-invariant `os.packages.install` core, the
  common `os.additionalFiles`, `os.services.enable` base, and the 7 shared `kernelCommandLine` entries
  ‚Äî all single-sourced in `image.yaml`'s `config:`. The verity variants **append** their extras
  (`rd.hostonly=0`, extra packages, `kdump`/`trident`) with no special syntax; release-divergent
  package names live in `by-release/` (the 4.0 section above).

## What did NOT decompose cleanly (honest findings) ‚öÝÔ∏è

- **Storage is per-variant, not shared.** The grub layout (esp 16M + single root) and the verity
  layouts (esp 512M + A/B + hash + var + overlay) are *fundamentally different*, so each variant owns
  a whole `storage` subtree via `$include`. A partial patch would be the wrong tool here ‚Äî the layouts
  share almost nothing ‚Äî and tailor's merge is intentionally generic (deep-merge maps, append lists,
  `$set`/`$replace`/`$remove`); it does **not** key lists by any IC field. This is the bulk of the
  remaining lines, and it is **irreducible**: the images really are that different.
- **`scripts` are owned whole per-variant.** `postCustomization` is **order-sensitive** and the
  variants interleave the shared and unique scripts differently, so splitting them into base+append
  would change run order. The whole list lives in each `by-variant` file (correct, but not deduped).
- **The `update` phase is nearly a separate image.** `updateimg-grub.yaml` has a different storage
  layout (5G single-root), a different grub package (`-noprefix`), extra files, and different scripts
  ‚Äî it is *not* a small delta on the base phase. It also only exists for `grub`/`grub-verity`. The
  honest recommendation is to model `update` as **its own image definition**, not a `phase` fragment
  (which is why `image.yaml`'s matrix is `phase: [base]` here).

## What this example demonstrates about the mechanism

1. **It's faithful** ‚Äî the fragments provably recompose the exact trident configs.
2. **The authoring is plain IC config** ‚Äî no `$select`, no logic in YAML; the path
   (`by-variant/root-verity.yaml`) is the condition. One `$include` pulls a shared layout.
3. **The render-and-verify loop catches real bugs.** While authoring `layouts/storage/root-verity.yaml`
   I wrote `options: defaults,ro` in flow style; the comma silently parsed as a key separator. The
   renderer's diff caught it immediately ‚Äî exactly the value the proposal's golden-diff/validation
   provides.
4. **It is honest about its limits** ‚Äî heterogeneous structure (verity storage, the update phase)
   does *not* magically deduplicate; the mechanism helps most with shared boilerplate and with the
   axes that multiply (arch/release/phase), not with genuinely-divergent images.
5. **A divergent release is cheap and safe.** Adding the Fedora-aligned 4.0 (different package names,
   a different rpm-source, a different base image, and an arch-specific grub package) touched **no**
   3.0 file and left every 3.0 golden byte-identical ‚Äî the `by-release/` + `params` split localized
   100% of the divergence.
6. **IC config is an opaque passthrough.** tailor merges and forwards `config:` verbatim ‚Äî it does
   **not** model IC's schema or gate IC versions. The verity variants enable UKI by writing both
   `os.uki.mode: create` and `previewFeatures: [uki]` *in their own fragment*; tailor passes those
   through unchanged (it never injects `previewFeatures` or validates them against an IC version).
   What IC accepts is the contract between the author and IC.
